Personal Skills

Skills that can be acquired by individual programmers

Learn to Debug

To see into the execution of a program by examining it

Primary skill for a programmer

A programmer who cannot debug is blind

Debugging is immense fun and a source of satisfaction

Debugging requires creativity and ingenuity

Never be afraid to debug and modify code


You are surrounded by Imperfect code

You don't understand why the code is failing?

Undocumented features?

Weird bugs and errors?

You must debug

How to Debug

  1. Use a debugging tool : Fine-grained control
  2. Printlining : Temporarily add lines that print program information
  3. Logging : Permanent Window into the program execution

Debugging tools are wonderful when they are available, But only avaiblable in dev environment

Printlining is quick and easy

Logging, when well designed and thorough, is absolutely wonderful!

Splitting the Problem Space

Divide and Conquer
  1. Can you design an experiment to isolate the code responsible for the crash?
  2. Narrow down the problem space by breaking it apart.

Debugging using a Log

Logs help you understand the many simple things as well as the overall complexity of a huge Software System

The information provided by Logs should be configurable

    Logs should -
  1. Provide information about bugs that are hard to reprodruce.
  2. Provide statistics and data relevant to performance.
  3. Allow debugging unanticipated problems whitout modifyng and redeploying code

Amount of logs should always be optimum. Too much logs, make it harder to find the required information, and too little logs may not have the required information

How to add Logging

    Each record in the log should identify
  • Position in source code
  • Precise time of execution
  • Thread executed if applicable
  • Some useful peice of information stored in a variable
  • An assigned level (TRACE, DEBUG, INFO, WARN, ERROR)


    Log statements should be sprinkled across the code, particularly around
  • Major Functionality
  • Risky Code

Dealing with Performance Problems

90% of the execution time is spent in 10% of the code

How to approach Performance Problems

    When you are faced with slow code :
  • First build a mental model.
  • Use a profiling tool or a good log to determine where the time and resources are spent
  • Usually expensive I/O is a prime suspect.

Fixing Performance Problems

    Performance of just released Software can be improved 10 to 100 times with little effort, because of all the short-cuts taken to meet deadlines
  • Key is to analyze the system and find the bottlenecks
  • Create experiments to easily reproduce the performance degradation at the bottlenecks
  • Target the low hanging fruit of bottlenecks first
  • Constantly test your fixes against the experiments to confirm performance improvements

Proposing a re-design should always be the last resort

Low hanging Fruit

    Low hanging fruit can be easily picked and provide an immediate benifit to the performance. Examples include
  • Failing to to provide a proper index to a DB table
  • Doing unnecessary I/O in inner loops
  • Failing to remove / deallocate objects when they are no longer required
  • Leaving debug statements that are not required
  • Wrong use of APIs for poorly documented external libraries.

Team Skills

Skills you need to work efficiently in a team a diverse characters

Estimation

I promise to do it in 5 weeks

I estimate that, if I really understand the problem, it is about 50% likely that we will be done in five weeks (if no one bothers us during that time).

Estimation is important

A Commercial Software System includes development, documentation, deployment, marketing, sales and finance.

Without predicting development time, it is impossible to plan the rest

Good estimation provides that predicability

So Managers love it

But Estimation is Hard

In fact it is impossible to estimate accurately

Developers often under-estimate

Managers often over-estimate

As long as both parties communicate often and state the assumptions, we minimize failures

Honest Estimation takes time and practice, you wont get it right the first time

On-the-spot estimations wont be accurate or honest

How to Estimate

Stall and prototype the task, if pressure permits

Pad extra-time to your estimates, explicitly

Implicityly padding extra-time to your estimates may also work, but will deplete trust within the team

State forcefully when you cannot estimate due to a big unknown risk

How to get Information from People

Ask Questions

    Ask a lot of Questions, but
  • Respect every person's time and balance it against your own
  • The value diminishes the more you do it.
  • Time is the most precious commodity a person has, and differs in value from person to person

Cost vs Benifit

    Everyone benifits from talking to you
  • They learn more about you, get a different perspective on a question and might enjoyyour presence
  • You will learn more about them and may learn the answer you seek
  • But the more they talk to you, the less benifit they derive

Judgement Skills

As a programmer, you need to make decisions everyday

Time Complexity

Vs

Space Complexity

You cannot become a better programmer without knowing the computational complexity theory

    You should understand
  • the big O notation
  • difference between constant time, nlog(n) and n squared

Basics

  • Time is the processor cycles used to execute a program
  • Space is the temporary memory used the program during execution
  • Time and Space can be traded off against each other
  • The amount of time it takes to run an algorithm depends on the size of the input
  • If it is has only one loop, going through all the elements of the input , the conplexity is n
  • If it is has nested loops, the conplexity is n squared

How do you do it

You can save time by maintaining more information, thereby reducing the computation required to calculate it

    Examples -
  • You can save space by encoding data more tighly, at the expense of spending more computing time to extract it
  • Caching is possible by taking up more space and replicating the data, but saves a lot of computation time
Using too much memory to save computation time, could also affect other programs jostling for the same memory. So always proceed with caution

Code Quality

Vs

Development Time

Sofware Development is always a compromise between what the project does and getting the project done

You are often offended when asked to sacrifice code quality to speed up development

How to handle it

  • First inform the rest of the team and quantify the cost of decrease of quality
  • Explain at what cost the lost ground will be regained in the next cylce
  • If this also affects the Testing efforts , point that out.
  • If it leads to more bugs reported after Testing is done, point that out
  • If you still have to do it, isolate the patch-up code into particular components, that can be easily re-written in the next cycle
  • Let your team know and document the patch up code clearly

Take Away

  • Good design will be resillient against poor code implementations
  • With good interfaces and abstractions, eventual rewrites will be far less painful.
